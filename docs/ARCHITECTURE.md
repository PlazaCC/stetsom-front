# Stetsom Front Architecture

Next.js 16 application serving the Stetsom institutional website and admin CMS. Consumes stetsom-api for all data.

---

## Overview

Two surfaces share a single Next.js codebase:

| Surface | Path | Auth |
|---|---|---|
| Public site | `src/app/[locale]/(site)/` | None |
| Admin CMS | `src/app/admin/` | Bearer JWT via HttpOnly cookie |

The Next.js server also acts as a BFF, proxying authenticated browser requests to stetsom-api via `src/app/api/`.

---

## Route Structure

```
src/app/
├── [locale]/                 Locale wrapper. pt-BR has no URL prefix.
│   └── (site)/               Public site with Header and Footer.
│       ├── page.tsx          Home page
│       ├── produtos/         Product listing and detail
│       ├── sobre/            About page
│       └── suporte/          Support page
├── admin/                    Admin CMS
├── api/
│   ├── bff/[...path]/        Proxy to stetsom-api
│   └── auth/                 Login, logout, and refresh
├── layout.tsx                Root layout. Loads globals.css.
└── globals.css
```

Route groups with `(name)` syntax do not add a URL segment. Directories prefixed with `_components/` are co-located with a route and not shared across routes.

## Data Layer

Types and hooks are generated by Orval from the stetsom-api OpenAPI spec. Never define API shapes manually.

- Types: `src/api/stetsom/model/` (barrel: `index.ts`)
- Hooks: `src/api/stetsom/endpoints/<tag>/<tag>.ts`
- Regenerate: `pnpm api:generate`

Two client entry points, selected by rendering context:

| Context | Client | Route |
|---|---|---|
| Client component / React Query | `orvalClient` from `@/api/stetsom/orval-client` | `/api/bff/*`. The BFF adds the Bearer token. |
| Server component / Route handler | `serverOrvalClient` from `@/api/stetsom/orval-server` | Direct to stetsom-api |

Never call `fetch()` directly for stetsom-api data.

---

## BFF Layer

`src/app/api/bff/[...path]` proxies requests to stetsom-api. It reads the `admin_token` HttpOnly cookie and forwards it as an `Authorization: Bearer` header. stetsom-api validates the JWT and roles.

Dedicated route handlers exist for operations the BFF cannot proxy generically:

| Route | Purpose |
|---|---|
| `api/auth/login` | Exchanges credentials for tokens, sets HttpOnly cookie |
| `api/auth/logout` | Clears the cookie |
| `api/auth/refresh` | Rotates the access token |
| `api/upload` | Presigns S3 upload intent with auth injection |

## Observability

Sentry runs only in Vercel Production. Without `NEXT_PUBLIC_SENTRY_DSN`, the client, Node.js, and Edge SDKs are not initialized, so local and Preview deployments send no telemetry.

The official Next.js integration captures browser errors, React root errors, Server Components, Route Handlers, and Edge failures. `sentry.server.config.ts` sets `tracePropagationTargets` to `CMS_API_BASE_URL`, so sampled traces link the full request path `browser → Next.js/BFF → stetsom-api` into one trace instead of disconnected transactions. Source maps and releases are managed by the Vercel integration.

Unhandled failures become Sentry Issues. Recovered upstream failures use `warn`/`error` logs, and expected responses such as `404` are not reported. Automatic collection of user identity, cookies, headers, bodies, query parameters, and stack-frame variables is disabled.

Session Replay and Profiling are intentionally not enabled anywhere in this project. Replay's free quota is small (50 sessions/month) and it captures session/DOM content, a bigger sensitive-data surface than error, trace, and log telemetry. Profiling requires pay-as-you-go on every Sentry plan tier, including Free.

## Auth Flow

```
1. POST /api/auth/login  { email, password }
   → stetsom-api returns { accessToken, refreshToken }
   → BFF stores accessToken as HttpOnly cookie "admin_token"

2. Protected browser requests
   → orvalClient calls /api/bff/*
   → BFF reads "admin_token", adds Authorization: Bearer
   → stetsom-api validates and enforces roles

3. Token refresh
   → POST /api/auth/refresh  { refreshToken }
   → new accessToken replaces the cookie
```

`admin_token` is HttpOnly and cannot be read by JavaScript. All cookie operations are server-side.

## i18n

Locales: `pt-BR` (default, no URL prefix), `/en`, `/es`. Configuration lives in `src/i18n/`.

API multilingual fields use `I18nString { pt: string, en?: string, es?: string }`. The backend locale key is `pt`, not `pt-BR`. Public responses are already resolved by stetsom-api.

Navigation links, `useRouter`, and `usePathname` must be imported from `@/i18n/navigation`, not from `next/link` or `next/navigation`.

## Mock Mode

> **Offline fallback:** Develop and PR previews use the dedicated API. Mock mode
> exists only for local work without API access and must remain contract-compatible.

Set `USE_MOCK_DATA=1` in `.env.local` to serve GET requests from local fixtures without a running API. Mutations return `{ _mock: true }`.

- Fixtures: `src/lib/mock/*.ts`, consumed by `src/lib/api/providers/mock-provider.ts`
- Mock payloads conform to Orval-generated types from `@/api/stetsom/model`
- Refresh: `pnpm mock:dump` (requires real API credentials in `.env.local`)

The mock provider is active by default when `CMS_API_BASE_URL` is unset (local dev without backend).

## Design System

Tailwind CSS v4 with no config file. All tokens are declared in `src/app/globals.css` inside `@theme inline`. Key token groups:

- Brand colors: `bg-brand`, `text-brand`, `bg-brand-dark`
- Typography scale: `text-display-*`, `text-section-title`, `text-button-md`, `text-2xs`
- Font families: `font-sans` (Barlow), `font-sans-condensed` (Barlow Condensed)

CMS admin styles are scoped under a `.cms` CSS class applied in `src/app/admin/layout.tsx`, with shadcn tokens remapped in `src/app/admin/admin.css`. Use the Figma MCP (`/refine-design` skill) to inspect design when needed.

---

## Component Architecture

- Server Component by default. Add `"use client"` only for event handlers, hooks, or browser APIs.
- Shared presentational primitives live in `src/components/ui/` and must not contain data hooks.
- Route-co-located sections live in `src/app/[route]/_components/` and may contain data hooks but are not shared across routes.
- `cn()` from `@/lib/utils` for conditional Tailwind class merging.
