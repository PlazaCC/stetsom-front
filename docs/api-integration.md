# API Integration

All data communication with stetsom-api goes through the Orval-generated data layer. Never call `fetch()` directly for API data.

For the complete architecture context, see `docs/ARCHITECTURE.md`.

---

## Data Layer

Types and React Query hooks are generated by Orval from the stetsom-api OpenAPI spec.

- Types: `src/api/stetsom/model/` (barrel: `index.ts`)
- Hooks: `src/api/stetsom/endpoints/<tag>/<tag>.ts`
- Regenerate: `pnpm api:generate`

Two client entry points, selected by rendering context:

| Context | Client | Route |
|---|---|---|
| Client component / React Query | `orvalClient` from `@/api/stetsom/orval-client` | `/api/bff/*` — BFF injects HttpOnly cookie |
| Server component / Route handler | `serverOrvalClient` from `@/api/stetsom/orval-server` | Direct to stetsom-api |

Hook naming: `useGetApi*` for reads, `usePostApi*` / `usePatchApi*` / `useDeleteApi*` for mutations. Include locale in `queryKey` for any hook returning locale-variant data.

## Auth

```
1. POST /api/auth/login  { email, password }
   → tokens stored as HttpOnly cookie "admin_token"

2. Protected requests
   → orvalClient → /api/bff/* → BFF adds Authorization: Bearer → stetsom-api

3. Token refresh
   → POST /api/auth/refresh  { refreshToken }  → new accessToken
```

`admin_token` is HttpOnly and unreadable from JavaScript. All cookie operations are server-side.

Public routes (no token needed): `/api/pages/*`, `/api/categories`, `/api/products`, `/api/contact`.

Protected routes: `/api/dashboard`, `/api/users`, `/api/banners`, `/api/library`, `/api/messages`, `/api/audit`, `/api/config`, `/api/upload/*`, `/api/products/admin*`.

## Error Contract

All API errors throw `OrvalApiError`. Use `toApiError(err)` to cast unknown throws.

```ts
err.status  // HTTP status number
err.code    // machine-readable identifier — branch on this
err.message // human-readable — display only, never branch on this
```

| Status | Code |
|---|---|
| 401 | `UNAUTHORIZED` |
| 404 | `NOT_FOUND` |
| 409 | `CONFLICT` |
| 422 | `VALIDATION_ERROR` |
| 503 | `SERVICE_UNAVAILABLE` |

## Pagination

| Style | Response shape | Query params | Endpoints |
|---|---|---|---|
| Page-based | `{ items, total, page, pageSize, totalPages }` | `page` (1-indexed), `pageSize` (max 100) | Products catalog, CMS products |
| Offset-based | `{ items, total }` | `limit` (max 200), `offset` | Users, messages, audit, library, banners |

## Upload Flow

Files are uploaded directly to S3. The API never receives the binary.

```
1. POST /api/upload  { fileName, mimeType, sizeBytes?, scope? }
   → { uploadUrl, file_url, headers, assetType, expiresIn }

2. PUT uploadUrl
   Headers: exactly the headers from step 1 — do not add Authorization
   Body: file binary

3. POST /api/upload/complete  { filename, file_url, type, size_bytes, ... }
   → { asset: LibraryAsset }
```

`file_url` from step 1 is the permanent public URL — store it in resource fields. `uploadUrl` expires in 15 minutes and must not be reused. The asset is not registered in the library until step 3 completes.

## Page Payloads

Public page routes (`/api/pages/home`, `/api/pages/about`, `/api/pages/catalog`, `/api/pages/support`) return a complete payload for the entire page in one request. Do not split into multiple fetches.

These endpoints return `503` when a content dependency is unavailable. Handle with a fallback state, not a 404 page.

## User Roles

The server enforces roles. The frontend uses them for UI decisions only.

| Role | Access |
|---|---|
| `SUPER_ADMIN` | Full access including user management |
| `ADMIN` | Content management, no user administration |
| `EDITOR` | Read and draft; no publish or destructive actions |

---

## Audit Log

Each `AuditEntry` includes `action_sentence` — a pre-formatted human-readable string. Render it directly without reformatting. Filter by entity with the `?entity=` query parameter.

Actions: `CREATE`, `UPDATE`, `DELETE`, `LOGIN`, `LOGOUT`, `PUBLISH`.
